# PR Review Toolkit

一套全面的专用 Agent 集合，用于深入审查 Pull Request，涵盖代码注释、测试覆盖率、错误处理、类型设计、代码质量和代码简化。

## 概述

该插件包含 6 个专家审查 Agent，每个 Agent 专注于代码质量的特定方面。可以单独使用它们进行针对性审查，或组合使用进行全面的 PR 分析。

## Agent

### 1. comment-analyzer
**专注领域**: 代码注释的准确性和可维护性

**分析内容**:
- 注释与实际代码的准确性对比
- 文档完整性
- 注释腐化和技术债务
- 误导性或过时的注释

**使用时机**:
- 添加文档后
- 在最终化包含注释更改的 PR 之前
- 审查现有注释时

**触发示例**:
```
"检查注释是否准确"
"审查我添加的文档"
"分析注释中的技术债务"
```

### 2. pr-test-analyzer
**专注领域**: 测试覆盖率质量和完整性

**分析内容**:
- 行为覆盖率与行覆盖率
- 测试覆盖率中的关键缺口
- 测试质量和韧性
- 边缘情况和错误条件

**使用时机**:
- 创建 PR 后
- 添加新功能时
- 验证测试全面性

**触发示例**:
```
"检查测试是否全面"
"审查此 PR 的测试覆盖率"
"是否存在关键的测试缺口？"
```

### 3. silent-failure-hunter
**专注领域**: 错误处理和静默失败

**分析内容**:
- catch 块中的静默失败
- 不当的错误处理
- 不合适的回退行为
- 缺失的错误日志

**使用时机**:
- 实现错误处理后
- 审查 try/catch 块时
- 在最终化包含错误处理的 PR 之前

**触发示例**:
```
"审查错误处理"
"检查静默失败"
"分析此 PR 中的 catch 块"
```

### 4. type-design-analyzer
**专注领域**: 类型设计质量和不变量

**分析内容**:
- 类型封装（1-10 分评级）
- 不变量表达（1-10 分评级）
- 类型实用性（1-10 分评级）
- 不变量强制执行（1-10 分评级）

**使用时机**:
- 引入新类型时
- 创建包含数据模型的 PR 时
- 重构类型设计时

**触发示例**:
```
"审查 UserAccount 类型设计"
"分析此 PR 中的类型设计"
"检查此类型是否有强不变量"
```

### 5. code-reviewer
**专注领域**: 项目规范的通用代码审查

**分析内容**:
- CODEBUDDY.md 合规性
- 风格违规
- Bug 检测
- 代码质量问题

**使用时机**:
- 编写或修改代码后
- 提交更改前
- 创建 Pull Request 前

**触发示例**:
```
"审查我最近的更改"
"检查一切是否正常"
"在我提交前审查此代码"
```

### 6. code-simplifier
**专注领域**: 代码简化和重构

**分析内容**:
- 代码清晰度和可读性
- 不必要的复杂性和嵌套
- 冗余代码和抽象
- 与项目标准的一致性
- 过于紧凑或聪明的代码

**使用时机**:
- 编写或修改代码后
- 通过代码审查后
- 代码可用但感觉复杂时

**触发示例**:
```
"简化此代码"
"使此代码更清晰"
"优化此实现"
```

**注意**: 此 Agent 在改进代码结构和可维护性的同时保持功能不变。

## 使用模式

### 单独使用 Agent

只需提出与 Agent 专注领域匹配的问题，CodeBuddy 会自动触发相应的 Agent：

```
"你能检查测试是否覆盖了所有边缘情况吗？"
→ 触发 pr-test-analyzer

"审查 API 客户端中的错误处理"
→ 触发 silent-failure-hunter

"我添加了文档 - 是否准确？"
→ 触发 comment-analyzer
```

### 全面 PR 审查

要进行深入的 PR 审查，可以请求多个方面的检查：

```
"我准备创建此 PR 了。请：
1. 审查测试覆盖率
2. 检查静默失败
3. 验证代码注释是否准确
4. 审查任何新类型
5. 通用代码审查"
```

这将触发所有相关 Agent 来分析 PR 的不同方面。

### 主动审查

CodeBuddy 可能会根据上下文主动使用这些 Agent：

- **编写代码后** → code-reviewer
- **添加文档后** → comment-analyzer
- **创建 PR 前** → 多个适当 Agent
- **添加类型后** → type-design-analyzer

## Agent 详情

### 信心评分

Agent 为其发现的问题提供信心评分：

**comment-analyzer**: 在准确性检查中高置信度识别问题

**pr-test-analyzer**: 对测试缺口进行 1-10 分评级（10 = 关键，必须添加）

**silent-failure-hunter**: 标记错误处理问题的严重性

**type-design-analyzer**: 对 4 个维度进行 1-10 分评级

**code-reviewer**: 对问题进行 0-100 分评分（91-100 = 关键）

**code-simplifier**: 识别复杂性并建议简化方案

### 输出格式

所有 Agent 提供结构化、可操作的输出：
- 清晰的问题识别
- 具体的文件和行引用
- 问题原因说明
- 改进建议
- 按严重性优先级排序

## 最佳实践

### 何时使用每个 Agent

**提交前**:
- code-reviewer（通用质量）
- silent-failure-hunter（如果更改了错误处理）

**创建 PR 前**:
- pr-test-analyzer（测试覆盖率检查）
- comment-analyzer（如果添加/修改了注释）
- type-design-analyzer（如果添加/修改了类型）
- code-reviewer（最终扫描）

**通过审查后**:
- code-simplifier（提高清晰度和可维护性）

**PR 审查期间**:
- 任何 Agent 用于特定关注点
- 修复后有针对性的重新审查

### 运行多个 Agent

可以请求并行或顺序运行多个 Agent：

**并行**（更快）:
```
"并行运行 pr-test-analyzer 和 comment-analyzer"
```

**顺序**（当一个影响另一个时）:
```
"先审查测试覆盖率，然后检查代码质量"
```

## 提示

- **具体明确**: 针对特定 Agent 进行重点审查
- **主动使用**: 在创建 PR 前运行，而非之后
- **优先处理关键问题**: Agent 会优先排序发现结果
- **迭代**: 修复后再次运行以验证
- **不要过度使用**: 专注于更改的代码，而非整个代码库

## 故障排除

### Agent 未触发

**问题**: 请求审查但 Agent 未运行

**解决方案**:
- 在请求中更加具体
- 明确提及 Agent 类型
- 引用具体关注点（例如"测试覆盖率"）

### Agent 分析了错误的文件

**问题**: Agent 审查了太多或错误的文件

**解决方案**:
- 指定要专注的文件
- 引用 PR 编号或分支
- 提及"最近的更改"或"git diff"

## 工作流集成

此插件可与以下内容良好配合：
- **build-validator**: 在审查前运行构建/测试
- **项目特定 Agent**: 与您的自定义 Agent 组合

**推荐工作流**:
1. 编写代码 → **code-reviewer**
2. 修复问题 → **silent-failure-hunter**（如果是错误处理）
3. 添加测试 → **pr-test-analyzer**
4. 编写文档 → **comment-analyzer**
5. 审查通过 → **code-simplifier**（打磨）
6. 创建 PR

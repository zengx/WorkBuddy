## 目标

根据参考节点分析并生成相应的测试场景，场景节点必须是参考节点的子节点。
请综合分析参考节点的功能特点，识别出需要验证的各种业务场景。

## 约束

1. 参考节点必须有效，否则直接报错
2. 只生成SCENE类型的场景节点
3. 生成的场景必须与参考节点功能相关
4. 每个场景必须有明确的测试目标和验证范围
5. 场景应覆盖正常流程、异常流程和边界条件

## 场景类型

根据测试需要，可生成以下类型的测试场景：

1. **正常业务场景** - 标准用户操作流程和预期行为
2. **异常处理场景** - 错误输入、系统异常和容错处理
3. **边界条件场景** - 极限值、临界状态和边界情况
4. **并发访问场景** - 多用户同时操作和资源竞争
5. **数据一致性场景** - 数据完整性和事务一致性验证
6. **性能压力场景** - 高负载、大数据量和响应时间验证
7. **安全防护场景** - 权限控制、数据安全和攻击防护
8. **集成交互场景** - 模块间交互和外部系统集成

## 场景设计原则

1. **业务完整性** - 每个场景应代表一个完整的业务操作
2. **独立可执行** - 场景之间相互独立，可单独执行
3. **可验证性** - 每个场景都有明确的验证标准
4. **覆盖全面性** - 涵盖功能的各个方面和使用情况
5. **优先级明确** - 根据业务重要性和风险级别划分优先级

## 场景生成策略

### 基于模块节点生成场景

- 分析模块的核心功能和业务流程
- 识别模块的输入输出和状态变化
- 考虑模块与其他模块的交互关系
- 覆盖模块的各种使用模式和异常情况

### 基于需求节点生成场景

- 解析需求的功能描述和验收标准
- 识别需求涉及的业务流程和用户操作
- 分析需求的约束条件和限制要求
- 考虑需求实现可能遇到的各种情况

## 场景命名规范

1. 场景名称应简洁明确，体现测试重点
2. 使用动宾结构，如"用户登录成功场景"
3. 区分不同类型场景，如"正常登录"、"密码错误登录"
4. 避免使用技术术语，优先使用业务术语
5. 保持命名风格一致性

## 输出要求

**🚨 关键规则 - 确保输出稳定性**：

1. **强制使用标准JSON格式输出**
   - 必须输出严格的JSON数组格式
   - 所有字段必须使用双引号
   - 特殊字符必须正确转义（引号用 `\"`，换行符用 `\n`）
   - 不允许使用YAML格式

2. **输出格式验证**
   - 输出必须是可直接解析的JSON数组
   - 每个场景节点必须包含所有必需字段：`name`, `description`, `kind`, `parent_uid`
   - 不生成 `uid` 字段（系统会自动分配）

3. **禁止行为**：
   - ❌ 不要创建任何 Python 脚本
   - ❌ 不要使用YAML格式输出
   - ❌ 不要在JSON中添加注释
   - ❌ 不要输出除JSON数组外的任何其它内容（除了简短确认信息）

4. 直接输出到临时文件中

## 字段说明

| 字段          | 类型   | 必填 | 说明                                      |
| ------------- | ------ | ---- | ----------------------------------------- |
| `name`        | string | ✅   | 场景名称，简洁明确                        |
| `description` | string | ✅   | 场景详细描述                              |
| `kind`        | string | ✅   | 固定值：`SCENE`                           |
| `parent_uid`  | string | ✅   | 必须使用 `ref_nodes` 参数中提供的节点 uid |

## 特殊字符处理规则

在JSON字符串中，以下字符必须转义：

- 双引号：`"` → `\"`
- 反斜杠：`\` → `\\`
- 换行符：不允许使用真实换行，用 `\n` 或用逗号/分号分隔
- 制表符：`\t`

**示例**：

```json
{
  "name": "\"正常\"登录场景",
  "description": "验证用户使用正确凭据成功登录系统的完整流程，包括：输入验证、身份认证、会话创建"
}
```

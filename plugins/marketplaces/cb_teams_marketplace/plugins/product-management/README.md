# 产品管理插件

产品管理插件覆盖完整的 PM 工作流程：编写功能规格、管理产品路线图、与利益相关者沟通、综合用户研究、分析竞品和审查产品指标。

## 安装

本插件为 CB Teams Marketplace 的一部分，通过团队配置自动安装。

## 核心能力

### 1. 功能规格与 PRD 编写

从问题陈述或功能想法生成结构化的产品需求文档。包括用户故事、需求优先级、成功指标和范围管理。

**适用场景**：
- 规划新功能或产品
- 从问题陈述创建 PRD
- 为工程团队记录需求
- 管理功能范围

**生成内容**：
- 问题陈述
- 目标和非目标
- 用户故事（按角色分组）
- 需求分类（P0/P1/P2）和验收标准
- 成功指标（领先和滞后指标）
- 未解决的问题
- 时间线考虑

### 2. 产品路线图规划

创建、更新和重新排列产品路线图优先级。支持 Now/Next/Later、季度主题和 OKR 对齐的格式，包含依赖关系映射。

**适用场景**：
- 创建新产品路线图
- 添加功能到现有路线图
- 重新排列路线图优先级
- 更新路线图项目状态
- 调整时间线或日期
- 沟通路线图变更

**支持的操作**：
- **添加项目**：新功能、计划或工作项
- **更新状态**：未开始、进行中、有风险、阻塞、已完成、已取消
- **重新排优先级**：使用 RICE、MoSCoW、ICE 等框架
- **移动时间线**：识别下游影响
- **创建新路线图**：选择时间框架和格式

### 3. 利益相关者更新

生成针对受众（高管、工程、客户、董事会）量身定制的状态更新。清晰传达进展、风险和决策。

**适用场景**：
- 周度团队更新
- 月度领导层更新
- 功能发布公告
- 临时升级或转向
- 董事会或投资者更新
- 跨职能伙伴更新

**按受众定制**：
- **高管/领导层**：高层次、结果导向、简洁（300字以内）
- **工程团队**：技术细节、实施背景、阻塞因素、需要的决策
- **跨职能伙伴**：适当的背景细节、关注共同目标和依赖关系
- **客户/外部**：以利益为中心、清晰的时间线、无内部术语
- **董事会**：指标驱动、战略性、风险聚焦、非常简洁

### 4. 用户研究综合

将访谈笔记、调查数据和反馈转化为结构化见解。识别主题、构建用户画像，并找出有支撑证据的机会领域。

**适用场景**：
- 用户访谈后
- 分析调查回复
- 综合支持工单或反馈
- 合并多个研究来源
- 需要识别主题和模式
- 创建用户画像或用户细分
- 识别机会领域

**输入类型**：
- 访谈笔记或录音摘要
- 调查回复
- 可用性测试结果
- 支持工单
- 销售电话笔记
- 产品分析数据

**输出内容**：
- 研究概览（方法、参数、时间框架）
- 关键发现（证据、频率、影响、信心水平）
- 用户细分/画像
- 机会领域
- 优先级建议
- 未解决的问题

### 5. 竞品分析

研究竞争对手并生成包含功能对比、定位分析和战略影响的简报。

**适用场景**：
- 分析特定竞争对手
- 跨市场的功能对比
- 定位和上市分析
- 销售支持/战斗卡
- 战略规划
- 投资者或董事会材料
- 识别竞争机会

**研究来源**：
- 产品页面和功能列表
- 定价和打包
- 产品发布、博客、更新日志
- 媒体报道和分析师报告
- 客户评论（G2、Capterra、TrustRadius）
- 招聘信息（战略方向信号）
- 社交媒体和社区讨论

**分析内容**：
- 竞争对手概览
- 功能对比矩阵
- 定位分析
- 优势和劣势
- 机会和威胁
- 战略影响

### 6. 产品指标审查

分析产品指标、识别趋势、对比目标，并提出有行动建议的见解。

**适用场景**：
- 定期指标审查（周度、月度、季度）
- 分析产品健康状况
- 调查指标趋势
- 为领导层或董事会审查做准备
- 发布后分析
- 识别关注领域

**分析框架**：
- 北极星指标
- L1 健康指标（获取、激活、参与、留存、收入、满意度）
- L2 诊断指标
- 目标对比（OKR）

**输出内容**：
- 执行摘要
- 指标记分卡（表格格式）
- 趋势分析（当前值、趋势、vs目标）
- 亮点和关注领域
- 推荐行动
- 背景和注意事项

## 使用场景示例

### 1. 编写产品需求文档

**场景**：需要为企业客户添加 SSO 支持

**提示词示例**：
```
帮我写一个功能规格文档，我们需要为企业客户添加 SSO（单点登录）支持。
目标用户是企业 IT 管理员和他们的员工。
成功指标是减少企业客户的登录支持工单，提高企业客户的采用率。
```

**产出**：完整的 PRD，包括问题陈述、目标/非目标、用户故事、需求优先级、成功指标和开放问题。

### 2. 路线图规划和重排优先级

**场景**：季度路线图需要添加新功能并重排优先级

**提示词示例**：
```
我有以下 Q2 路线图项目：
1. SSO 集成（进行中）
2. 移动端优化（未开始）
3. 数据导出功能（未开始）
4. API 文档改进（未开始）

现在客户强烈要求优先支持 Webhooks。帮我评估如何调整优先级，并使用 RICE 框架分析。
```

**产出**：RICE 评分对比、重排优先级后的路线图、影响分析、建议的调整方案。

### 3. 准备利益相关者更新

**场景**：周度高管更新

**提示词示例**：
```
帮我准备本周的高管更新。
进展：完成了 SSO 集成的后端部分，前端开发进行中。
阻塞因素：等待安全团队审查 OAuth 实现。
下一步：完成前端集成，进行企业客户 Beta 测试。
```

**产出**：简洁的高管更新（300字以内），包含 TL;DR、状态（G/Y/R）、关键进展、风险缓解和具体要求。

### 4. 综合用户研究

**场景**：综合 8 次用户访谈的笔记

**提示词示例**：
```
我进行了 8 次用户访谈，关于他们使用我们产品的数据导出功能的体验。
以下是访谈笔记：
[粘贴访谈笔记]

请帮我综合主要发现、识别模式，并提出改进建议。
```

**产出**：结构化的研究综合，包含关键发现（带证据和频率）、用户画像、机会领域和优先级建议。

### 5. 竞品分析

**场景**：对比我们的 onboarding 流程和两个竞争对手

**提示词示例**：
```
请帮我分析我们的 onboarding 流程与 [竞争对手 A] 和 [竞争对手 B] 的对比。
重点关注：首次使用体验、设置步骤数量、Time to Value、用户反馈。
```

**产出**：竞品分析简报，包含功能对比表、优势/劣势分析、机会/威胁评估和战略建议。

### 6. 产品指标审查

**场景**：月度指标审查

**提示词示例**：
```
以下是我们上个月的关键指标：
- 月活用户（MAU）：15,200（上月：14,500，目标：16,000）
- 激活率：62%（上月：58%，目标：65%）
- 留存率（D7）：45%（上月：48%，目标：50%）
- 收入：$125K（上月：$118K，目标：$130K）

上个月我们发布了新的 onboarding 流程。请帮我审查这些指标。
```

**产出**：指标审查报告，包含趋势分析、亮点和关注领域、因果关系推测、推荐的行动项。

## 数据输入方式

此插件设计为**无需外部工具连接**即可工作，接受以下输入方式：

1. **粘贴文本**：直接粘贴文档内容、笔记、数据
2. **上传文件**：支持各种文档格式（.txt、.md、.docx、.pdf、.csv等）
3. **截图**：粘贴截图或图片
4. **口头描述**：用自然语言描述情况
5. **Figma 集成**（可选）：如配置了 Figma connector，可拉取设计稿和设计系统组件

## 最佳实践

### PRD 编写
- ✅ 范围要紧凑、明确，避免宽泛模糊
- ✅ 成功指标要具体可衡量
- ✅ 非目标和目标同样重要（防止范围蔓延）
- ✅ 未解决的问题要真实存在（不是你能回答的问题）

### 路线图管理
- ✅ 路线图是沟通工具，不是项目计划（保持在正确的抽象层次）
- ✅ 重排优先级前，先问"什么改变了"（基于新信息，而非心血来潮）
- ✅ 及早标记容量问题
- ✅ 明确显示依赖关系
- ✅ 添加新项目时，问什么要下移或移除（路线图是零和游戏）

### 利益相关者更新
- ✅ 不要埋没重点（从最重要的事开始）
- ✅ 状态颜色（绿/黄/红）要反映现实，不是乐观主义
- ✅ 要求要具体可执行
- ✅ 有坏消息时，先说坏消息
- ✅ 长度匹配受众注意力

### 用户研究综合
- ✅ 让数据说话（不要强行套入预定叙事）
- ✅ 区分用户说的和做的（行为数据比陈述偏好更可靠）
- ✅ 大量使用引用作为证据
- ✅ 明确标注信心水平
- ✅ 建议要具体到可执行

### 竞品分析
- ✅ 诚实评估竞争对手的优势
- ✅ 关注对客户重要的事，不是对产品团队重要的事
- ✅ 客户评论是金矿
- ✅ 战略影响部分是最有价值的（不要跳过）
- ✅ 注明分析日期（竞品分析有保质期）

### 指标审查
- ✅ 从"所以呢"开始（最重要的事是什么）
- ✅ 绝对数字无用，总是显示对比
- ✅ 小心归因（相关不等于因果）
- ✅ 细分分析常常揭示聚合指标掩盖的重要差异
- ✅ 如果审查没有导致至少一个行动，就是没用的

## 目标用户

- 产品经理
- 产品负责人
- 项目经理
- 创始人/CEO（早期产品规划）
- 工程经理（需要编写技术 PRD）
- 用户研究员（研究综合）
- 数据分析师（指标审查）

## 注意事项

- 本插件提供最佳实践指导和结构化输出，但不替代专业判断
- 竞品分析基于公开信息，可能不完全准确
- 用户研究综合取决于输入质量和样本规模
- 指标分析提供归因假设，需要进一步验证
- 所有输出应视为起点，需根据具体情况调整

# Schema 设计准则 (schema_principle.md)

> 本文件是主代理在 Step 3 设计 `schema.md` 时的强约束清单。目标不是枚举所有行业模板，而是让主代理用少量通用原则推断任意用户场景，产出可用、清晰、可维护的 Excel 设计图纸。
> 如果用户明确要求与本准则冲突，以用户要求为准，但仍应尽量保持可编辑、可计算、可阅读。

---

## 0. 设计目标

生成的 workbook 必须优先满足以下目标：

1. **能直接用**：用户打开后知道在哪里填写、在哪里看结果、哪些内容会自动计算。
2. **结构清楚**：输入、明细、计算、汇总、看板、备注等区域职责分明。
3. **公式联动**：能由输入推导的结果不要写死；汇总、合计、进度、差异、余额、达成率等优先设计为公式。
4. **视觉克制**：颜色用于区分层级和状态，不用于堆装饰；同一 sheet 主色不超过 2 个，警示色只用于异常或重点。
5. **便于维护**：普通数据表少合并单元格，字段命名短而稳定，公式逻辑写清楚。
6. **不过度设计**：不主动添加用户没有暗示的复杂 sheet、图表、透视表或宏。

## 1. 场景原型识别

主代理设计 schema 前，必须先判断用户需求命中的一个或多个**场景原型**。不要按行业硬套模板，要按工作方式组合原型。

| 原型 | 用户信号 | 默认设计方向 |
|---|---|---|
| 记录型 | 记录、清单、台账、明细、登记、列表、管理 | 单个明细表优先，字段规范、筛选、冻结表头 |
| 计划型 | 计划、安排、排期、里程碑、周期、负责人、交付 | 事项/任务 + 时间 + 负责人 + 状态 + 备注 |
| 打卡/进度型 | 打卡、完成、进度、学习、习惯、训练、检查项 | 状态列、周期列、完成统计、进度百分比 |
| 统计型 | 统计、汇总、分析、报表、排名、占比、趋势 | 明细 + 汇总，汇总引用明细，必要时加图表 |
| 表单/打印型 | 打印、签字、提交、填报、申请、考勤、登记表 | 标题区、填写区、签字栏、稳定版式、打印设置；长文本数据行避免统一固定行高 |
| 看板型 | 看板、一眼看、概览、KPI、仪表盘、领导汇报 | KPI 区 + 图表区 + 支撑明细，信息密度适中 |
| 日历/排程型 | 日历、月历、周计划、日程、排班、日期格 | 日期视图 + 事项清单，必要时按月/周拆分 |
| 计算模型型 | 预算、资产、收益、成本、库存、收支、测算 | 输入假设 + 计算区 + 结果区，公式必须可追踪 |

### 1.1 原型组合规则

- 一个需求可命中多个原型。例如"项目预算跟踪表" = 计划型 + 统计型 + 计算模型型。
- 命中统计型或看板型时，通常需要 `明细` + `汇总` 或 `看板` + `明细`。
- 命中表单/打印型时，可以使用合并标题、签字栏和固定版式；其他原型默认少用合并单元格。
- 用户只说"做个表/模板"且无其他线索时，默认按记录型轻量模板处理，不主动复杂化。

## 2. 结构决策

### 2.1 Sheet 拆分原则

| 条件 | 策略 |
|---|---|
| 逻辑单一、字段少、数据量 ≤ 50 行 | 单 sheet |
| 用户需要持续录入明细 | 建 `明细` 或业务名明细表 |
| 存在汇总、统计、看板、图表 | 至少拆为 `汇总/看板` + `明细` |
| 存在明显周期维度（月/周/季度）且每期版式相同 | 可按周期拆 sheet，或单 sheet 加周期字段；日历/排程型可按月拆 |
| 存在输入假设与计算结果 | 拆为 `输入` + `计算/明细` + `结果/汇总`，简单场景可放同一 sheet 的不同区域 |
| 用户明确指定 sheet 结构 | 严格遵循用户要求 |

- 单个 workbook 默认 sheet 数量 ≤ 10；超过时优先合并维度或询问是否需要拆文件。
- Sheet 名 ≤ 31 字符，使用简短中文或英文，禁止 `\ / * ? [ ] :`。
- 本生成链路用逗号传递 sheet 名，Sheet 名还应避免英文逗号 `,`。
- Sheet 顺序默认为：`看板/汇总` 在前，`明细/输入` 在后，`配置/说明` 最后。

### 2.2 区域模块

schema 中应按需要组合以下模块，而不是死套行业模板：

| 模块 | 用途 | 样式与行为 |
|---|---|---|
| 标题区 | 标明 workbook/sheet 主题 | 可合并居中；普通数据表只在首行使用 |
| 输入区 | 用户填写原始数据或假设 | 浅色底；避免公式；必要时数据验证 |
| 明细区 | 存放可筛选、可追溯的行级数据 | 表头固定，数据从下一行开始，少合并 |
| 计算区 | 放公式派生字段或中间计算 | 中性色或浅灰底；备注写清公式逻辑 |
| 汇总区 | 合计、平均、占比、达成率、余额等 | 强调色或加粗；必须引用明细/输入 |
| 状态区 | 完成、逾期、风险、异常、优先级 | 枚举 + 条件格式 |
| 时间区 | 日期、月份、周次、阶段、日历格 | 日期格式统一，可按周期分组 |
| 备注区 | 长文本、说明、下一步动作 | 宽列，自动换行，左对齐 |
| 打印区 | 部门、负责人、签字、填报日期 | 仅表单/打印型默认启用 |

### 2.3 表头与数据区

- 明细表表头默认在第 1 行；如有标题区，表头可在第 2 或第 3 行，但 schema 必须说明。
- 每列必须有表头，禁止数据列中间穿插空列。
- 数据从表头下一行开始，普通数据表禁止用空行做分组分隔。
- 如需汇总行，放在数据区最下方，或单独放汇总区，并在 Notes 中说明。
- 写计算列备注前，必须先确定该 sheet / 区块的行号锚点：标题行、表头行、数据起始行、数据结束行、合计行。Columns 表中的公式坐标必须由这些锚点推导，不得先写默认第 2 行公式再在 Notes 中另行改变布局。
- 如果 Notes 写了标题行或合计行，Columns 表备注必须同步使用真实数据行。例如 Notes 写"第 1 行标题、第 2 行表头、第 3~5 行数据、第 6 行合计"，则计算列备注应写 `B3`、`B3:B5`、`B6`，不能写 `B2:B4`。

## 3. 字段设计

### 3.1 类型系统

schema Columns 表中"类型"列必须描述单元格最终值 / 显示格式，使用以下标准类型之一：

| 类型 | 含义 | 子代理格式指令 |
|---|---|---|
| `文本` | 纯文字 | 左对齐 |
| `数字` | 整数或小数 | 右对齐，千位分隔符 |
| `数字（N位小数）` | 指定精度 | 右对齐，`0.00` 等 |
| `货币（¥）` / `货币（$）` | 金额 | 右对齐，`¥#,##0.00` 或 `$#,##0.00` |
| `百分比` | 0~1 的小数显示为百分比 | 右对齐，`0.00%` |
| `日期` | 日期 | `YYYY-MM-DD` |
| `日期时间` | 日期+时间 | `YYYY-MM-DD HH:mm` |
| `布尔` | 是/否、完成/未完成 | 居中；可用 `是/否`、`✓/✗` 或 `☑/☐` |
| `枚举` | 有限选项集 | 备注列列出全部选项 |
| `序号` | 自增行号 | 居中，纯整数 |

不得新增临时类型。单位不要写进类型列，应写在字段名或备注中，例如 `销售额（万元）` + `数字（2位小数）`。

Columns 表的"来源"列描述字段值从哪里来，也可描述字段值如何生成，例如 `用户填写`、`用户原话`、`推断`、`reference/<stem>/content.md`、`计算列`。

`计算列` 是字段的**生成方式**，不是推荐的显示类型。新写 schema 时：

- 如果字段由公式派生，Columns 表的"来源"列写 `计算列`。
- "类型"列仍写最终显示格式，例如 `百分比`、`货币（¥）`、`数字（2位小数）`、`日期`。
- "备注"列是计算列公式的权威位置：写完整公式或生成规则、引用字段/列、适用范围或填充方式、必要的空值 / 除零保护和格式说明。
- 公式坐标必须匹配字段所在 sheet / 区块的真实布局；若有标题行、多区块、合计行或表头不在第 1 行，备注必须写具体单元格和范围，例如 `B3`、`B3:B6`、`B7`，不得默认数据从第 2 行开始。
- 涉及除法的公式必须在公式本身显式包含分母空值和 0 值保护，例如 `OR(E2="",E2=0)` 或等价 `IFERROR`；达成率、同比、占比等比率公式若分子为空也应返回空白。不要只写"除零保护"。

### 3.2 列排布原则

- 默认顺序：**标识列**（序号、ID、名称）→ **时间列** → **维度列**（分类、区域、负责人）→ **状态列** → **度量列**（金额、数量、百分比）→ **计算列** → **备注列**。
- 强相关字段相邻放置，如 `单价`、`数量`、`小计`；`开始日期`、`结束日期`、`持续天数`。
- 字段名 ≤ 10 字为佳；单位可放字段名括号中，如 `金额（元）`。
- 单 sheet 列数 ≤ 26 为佳；超过时考虑拆 sheet、拆模块或精简字段。
- 备注、说明、行动项等长文本列放最后，设置较宽列宽和自动换行。

### 3.3 常见字段推断

| 原型 | 建议字段 |
|---|---|
| 记录型 | 序号、日期、类别、对象/名称、数量/金额、负责人、状态、备注 |
| 计划型 | 序号、任务、负责人、开始日期、结束日期、阶段、优先级、状态、交付物、备注 |
| 打卡/进度型 | 序号、项目、周期/日期、目标次数、完成状态、完成次数、完成率、备注 |
| 统计型 | 维度字段、指标字段、目标值、实际值、差异、达成率、排名/占比 |
| 表单/打印型 | 填报单位、填报人、日期、明细项、签字确认、备注 |
| 看板型 | 指标名称、本期值、目标值、差异、趋势、状态、说明 |
| 日历/排程型 | 日期、星期、事项、开始时间、结束时间、负责人、进度、备注 |
| 计算模型型 | 输入假设、数量、单价、金额、税率/费率、合计、差异、结果 |

## 4. 公式联动

### 4.1 基本原则

- 能由其他字段推导的值，"来源"必须标记为 `计算列`，不要写死。
- 汇总 sheet 的指标必须引用明细 sheet 或输入区，不要孤立手填。
- 模板类表格的公式应能在用户后续填入数据后自动更新。
- 计算列的公式本体必须写在 Columns 表的"备注"列，作为唯一权威来源；`Notes for sheet-agent` 不要重复另一版公式。
- 计算列备注必须按真实布局写公式坐标、适用范围和填充方式；普通明细表可写 `H2`、`H2:H25`，多区块 / 汇总表必须写该区块实际坐标，不能套用默认第 2 行。
- 简单聚合优先用 Excel 原生公式，如 `SUM`、`AVERAGE`、`COUNTIF`、`SUMIF`、`COUNTIFS`。
- 公式要按运算类型处理空值和除零。加减法、差异、余额等公式只需要保护必要空值，不应把合法的 0 值当作空白；涉及除法、排名、达成率、同比、占比时，Columns 表"备注"中的公式必须显式包含分母空值和 0 值保护，且比率公式若分子为空也应返回空白。Notes 只要求按 Columns 备注执行并读回验证公式文本。

### 4.2 必须优先公式化的字段

| 字段/场景 | 公式逻辑 |
|---|---|
| 小计/金额 | `单价 × 数量` |
| 合计/总计 | 对明细金额或数量求和 |
| 差异/偏差 | `实际值 - 目标值` 或 `本期 - 上期` |
| 达成率/完成率 | `实际值 / 目标值` 或 `完成数 / 总数` |
| 同比/环比 | `本期 / 对比期 - 1` |
| 余额 | `期初 + 收入 - 支出` |
| 出勤/完成次数 | 对 `√`、`☑`、`已完成` 等状态计数 |
| 持续天数 | `结束日期 - 开始日期 + 1` |
| 星期/月/周 | 由日期自动推导 |
| 热量/预算/库存收支 | `收入/摄入/入库 - 支出/消耗/出库` |

### 4.3 公式与样例数据

- 如果生成样例数据，样例数据必须与公式结果自洽。
- 如果用户要空模板，保留公式列，但公式应在输入为空时显示空白或 0，不显示错误值。
- 跨 sheet 公式要在 schema 中说明来源 sheet 和引用逻辑，例如"汇总!销售额合计引用 明细 的销售额列求和"。

### 4.4 汇总表坐标示例

有标题行的汇总 sheet 必须按真实行号写 Columns 备注。示例：

```markdown
### Columns
| 列 | 字段名 | 类型 | 来源 | 备注 |
|---|---|---|---|---|
| A | 渠道 | 文本 | 推断 | 第 3~5 行为渠道数据；A6 为合计 |
| B | 目标销售额合计（万元） | 数字（2位小数） | 计算列 | B3：`=SUMIF(明细!B:B,A3,明细!D:D)`；填充 B3:B5；B6：`=SUM(B3:B5)` |
| C | 实际销售额合计（万元） | 数字（2位小数） | 计算列 | C3：`=SUMIF(明细!B:B,A3,明细!E:E)`；填充 C3:C5；C6：`=SUM(C3:C5)` |
| F | 平均达成率 | 百分比 | 计算列 | F3：`=IF(OR(B3="",B3=0),"",C3/B3)`；填充 F3:F5；F6：`=IF(OR(B6="",B6=0),"",C6/B6)` |

### Notes for sheet-agent
- 第 1 行：标题，合并居中。
- 第 2 行：表头。
- 第 3~5 行：渠道数据。
- 第 6 行：合计行。
```

错误示例：上述 Notes 已声明第 3~5 行是数据时，Columns 表仍写 `B2：...；填充 B2:B4`。

## 5. 数据与样例策略

### 5.1 样例数据规模

| 用户意图 | Sample Data Scale 写法 |
|---|---|
| 明确说要 N 行数据 | `N 行` |
| 说"模板/给我填/空表" | `0 行（仅表头和公式/验证，留空给用户填写）` |
| 说"示例/样例/演示" | `5~10 行合理样例` |
| 没说清楚 | `5 行样例数据`，除非更像空模板 |
| 有附件且附件含真实数据 | `按 reference/<stem>/content.md 实际数据量` |
| 表单/打印型 | 按一页可打印容量设计，不为凑数据硬造大量行 |

### 5.2 样例数据质量

- 数据必须逻辑自洽，公式结果能对上。
- 数字分布要真实，不要全是整十整百。
- 日期要连续或符合业务周期；涉及工作日/自然月时要合理。
- 枚举值至少覆盖主要选项一次。
- 中文场景用中文样例数据；英文用户需求可用英文。
- 不主动造敏感个人信息、真实公司名、证件号、手机号等。

### 5.3 空值处理

- 不主动造空值，除非场景合理，如备注列可空。
- 附件或用户原始数据有空值时，在 Notes 中说明处理策略：留空、填 `N/A`、填 0 或排除统计。

## 6. 样式层级

### 6.1 样式总原则

- 同一 sheet 建议使用 1 个主色 + 1 个浅底色 + 必要警示色。
- 标题、表头、输入区、计算区、汇总区、状态区必须能视觉区分。
- 不使用大面积高饱和背景色；深色只用于标题/表头/关键汇总。
- 有真实数据的普通表格优先保证筛选和阅读，不为了美观破坏数据结构。

### 6.2 预设配色方案

| 风格 | 适用场景 | 表头背景 | 表头字色 | 输入/浅底 | 汇总/强调 |
|---|---|---|---|---|---|
| 商务蓝（默认） | 正式报告、经营分析、汇报 | `#4472C4` | `#FFFFFF` | `#D9E2F3` | `#2F5597` + 白字 |
| 轻量灰 | 自用、极简、数据清单 | `#F2F2F2` | `#1A1A1A` | `#FAFAFA` | `#E0E0E0` |
| 清新绿 | 项目、任务、进度、打卡 | `#70AD47` | `#FFFFFF` | `#E2EFDA` | `#548235` + 白字 |
| 日历多彩 | 日历、计划、年度目标 | 每个周期一个主色 | `#FFFFFF` | 主色 10%~20% 浅底 | 主色深一档 |
| 打印黑白 | 打印、签字、填报 | `#FFFFFF` | `#000000` | `#FFFFFF` | 粗边框/加粗，不依赖颜色 |
| 用户指定 | 用户明确提及颜色/风格 | 按用户要求 | 按用户要求 | 按用户要求 | 按用户要求 |

选择规则：

- 用户未提及风格，默认商务蓝。
- 用户说"简洁/简单/轻量/自用/极简"，用轻量灰。
- 用户说"项目/任务/进度/打卡/看板"，优先清新绿。
- 用户说"日历/月历/年度计划/排程"，可用日历多彩，但每个 sheet 内仍保持克制。
- 用户说"打印/签字/表单/考勤/登记"，优先打印黑白。
- 用户给出具体颜色，按用户指定。

### 6.3 表头、输入区、计算区

- 表头：加粗、水平居中、下边框 `thin`，按配色方案设置背景和字色。
- 输入区：浅底色，普通字体，明确留给用户填写。
- 计算区：浅灰或浅黄底，备注说明公式，不应让用户手填覆盖。
- 汇总区：加粗，上边框 `thin`，必要时深色背景 + 白字。
- 布尔/打卡列：居中，可用 `☐/☑` 或 `是/否`。
- 长文本列：左对齐、自动换行（wrapText）、列宽 200~300 px。

### 6.4 合并单元格

- 普通数据表、明细表默认不合并单元格，避免影响筛选、排序和复制。
- 允许合并的场景：标题区、打印表单抬头、签字栏、日历月标题、项目计划中的视觉分组。
- 使用合并单元格时，schema 必须说明这是版式需要；不要把合并用于普通数据分组。

---

## 7. 条件格式与高亮

### 7.1 触发条件

满足以下任一条件时，应在 Notes 中加入条件格式：

- 用户明确说"标红/高亮/预警/醒目/一眼看出"。
- 有状态列，如 `已完成/进行中/逾期/风险`。
- 有阈值判断，如达成率、预算执行率、库存下限、热量超支。
- 有日期逾期或临近截止日期。
- 有负数、亏损、异常值。

### 7.2 写法规范

必须写清三要素：**应用范围 + 条件 + 样式**。

示例：

```markdown
条件格式：
- 达成率列：>=100% → 绿底(#C6EFCE)+绿字(#006100)；<80% → 红底(#FFC7CE)+红字(#9C0006)
- 状态列："已完成"→ 绿底(#C6EFCE)；"逾期"→ 红底(#FFC7CE)；"进行中"→ 黄底(#FFEB9C)
- 截止日期列：日期 < 今天且状态 != "已完成" → 红字(#9C0006)
```

### 7.3 常用规则

| 场景 | 条件 | 样式 |
|---|---|---|
| 达标/超标 | 数值 >= 目标 | 绿底 `#C6EFCE` + 绿字 `#006100` |
| 低于预警线 | 数值 < 阈值 | 红底 `#FFC7CE` + 红字 `#9C0006` |
| 接近预警 | 阈值附近 | 黄底 `#FFEB9C` + 深黄字 `#9C6500` |
| 状态完成 | = "已完成" 或 `☑` | 绿底或绿字 |
| 状态逾期/风险 | = "逾期"/"高风险" | 红底或红字 |
| 日期逾期 | 日期 < 今天 | 红字 `#9C0006` |
| 负值/亏损 | 数值 < 0 | 红字 `#9C0006` |

不触发条件格式的场景：

- 纯空模板且没有状态/阈值。
- 用户只是要简单记录表，没有好坏判断。
- 用户明确说不要颜色或不要高亮。

## 8. 可用性规则

### 8.1 冻结、筛选、列宽

- 明细数据 ≥ 10 行，默认开启筛选。
- 明细数据 ≥ 20 行，默认冻结表头；有序号/ID/名称列且横向较宽时，同时冻结首列。
- 日期、金额、百分比必须使用统一格式。
- 表头和关键数字不得被截断。

**列宽策略**：

> 标注列宽时统一使用 **px** 单位（如 `280px`）。
> 子代理具备根据内容自主判断合理列宽的能力，主代理**只需标注宽列，普通列一律不指定**。

**① 主代理必须在 Notes 中标注像素值的列（仅限以下类型）：**

| 列类型 | 建议像素宽度 | 说明 |
|---|---|---|
| 长文本（任务描述、说明） | 200~260 px | 搭配 wrapText，单行约 12~16 汉字 |
| 备注/行动项 | 240~360 px | 搭配 wrapText，宽表可更大 |
| 地址/长路径 | 260~360 px | 搭配 wrapText |

写法示例：`F 列（家庭住址）列宽 280px，自动换行`

兜底规则：如果主代理无法判断某列是否属于长文本/备注/地址/长路径，但该列字段名或用户语义暗示会承载自由文本、多句描述、URL、文件路径、详细说明、处理意见、跟进记录等内容，应倾向于按宽列标注列宽和自动换行。

**② 以下类型全部交由子代理自行判断，主代理不得在 Notes 中为其指定像素值：**

序号、姓名、日期、枚举、状态、金额、百分比、手机号、邮箱、电话等标准类型列。子代理会根据表头文字长度和数据内容宽度自动选取合理宽度。

**③ 硬约束：**

- 长文本、备注类列**必须同时开启自动换行（wrapText）**。
- 主代理**禁止**在 Notes 中写出小于 60 的像素值。

### 8.2 数据验证

以下场景应设计数据验证或枚举说明：

- 状态列：`未开始/进行中/已完成/逾期/暂停`。
- 优先级列：`高/中/低`。
- 布尔列：`是/否` 或 `☐/☑`。
- 类别、部门、区域、负责人等可枚举字段。

如果工具能力无法创建真实下拉，至少在 Columns 备注列列出全部选项。

### 8.3 打印设置

命中表单/打印型时，在 Notes 中加入打印规则：

- 页面方向：列数多时用横向；窄表用纵向。
- 标题居中，可合并；保留部门、填报人、负责人、日期、签字确认等字段。
- 含长文本列并开启自动换行（wrapText）的数据区，schema Notes 中不要指定或建议数据行高；只有用户明确要求固定数据行高或每条记录等高时，才可指定数据行高，且必须保证内容不被裁切。
- 尽量一页容纳主表；如果数据多，说明按页延续。
- 打印型可用边框增强结构；颜色不应是唯一的信息载体。

## 9. 图表与看板

### 9.1 何时加图表

满足以下任一条件，且数据适合可视化时，应设计图表：

- 用户说"分析/趋势/对比/汇报/看板/KPI"。
- 存在时间序列、分类对比、占比构成或目标达成。
- 汇总 sheet 中有 3 个以上可比较指标。

用户只是要空模板、登记表、打印表单时，不主动加图表。

### 9.2 图表类型

| 数据特征 | 推荐图表 |
|---|---|
| 时间序列趋势 | 折线图 |
| 分类对比 | 柱状图（≤ 7 类）或条形图（> 7 类） |
| 占比/构成 | 饼图（≤ 5 类）或环形图 |
| 目标 vs 实际 | 组合图或柱状图 |
| 两个度量关系 | 散点图 |
| KPI 概览 | 指标块 + 小型趋势图 |

### 9.3 图表放置

- 默认放在汇总/看板 sheet。
- 简单明细表可把图表放在数据右侧空白区域。
- 图表必须引用表格数据区域，不要孤立造图。
- 在 schema 的 `Charts / Pivots` 段标注：所在 sheet、图表类型、数据源、表达目的。

## 10. 场景模块组合示例

以下不是固定模板，只是帮助主代理推断结构：

| 用户需求 | 原型组合 | 建议结构 |
|---|---|---|
| 做一份客户跟进表 | 记录型 + 计划型 | `客户跟进`：客户、联系人、阶段、下次跟进日期、负责人、状态、备注 |
| 做一份项目计划表 | 计划型 + 打卡/进度型 | `项目计划`：任务、衡量标准、交付物、周期、权重、周计划、状态 |
| 做一份销售汇总表 | 记录型 + 统计型 + 看板型 | `汇总` + `明细`，汇总引用明细，含销售额、目标、达成率、趋势图 |
| 做一份考勤表，要能打印 | 表单/打印型 + 统计型 | 标题区、部门/负责人、日期列、姓名行、出勤合计公式、签字栏、横向打印 |
| 做一份学习进度表 | 打卡/进度型 + 计划型 | 章节、完成状态、计划日期、完成日期、完成率、备注，状态高亮 |
| 做一份预算跟踪表 | 计算模型型 + 统计型 | `预算明细` + `汇总`，预算、实际、差异、执行率，超支标红 |
| 做一份年度/月度日历 | 日历/排程型 + 计划型 | 月份 sheet 或月度分区，日历视图 + 事项清单 + 备注区 |

## 11. schema.md 编写要求

### 11.1 必填内容

主代理写出的 `schema.md` 必须包含：

- User Intent：用 1~2 句复述用户需求。
- Scenario Archetype：列出命中的场景原型及理由。
- Sheets：列出每个 sheet 的角色。
- 每个 sheet 的 Columns 表：列字母、字段名、类型、来源、备注。
- Sample Data Scale：说明数据规模策略。
- Notes for sheet-agent：写清样式、条件格式、冻结/筛选/打印等执行要求；公式只引用 Columns 表备注，不重复写另一版公式。
- 约束与不做的事：明确本次不做哪些扩展，防止过度发挥。

### 11.2 Notes 写法

Notes 必须具体，不能写模糊词。错误写法："做得美观一点"。正确写法：

```markdown
Notes for sheet-agent:
- 表头：商务蓝 `#4472C4` 底 + 白字 + 加粗 + 居中 + 下边框 thin
- 输入区：浅蓝 `#D9E2F3` 底；用户填写列不放公式
- 计算列：所有 `来源=计算列` 的字段，按 Columns 表"备注"中的公式 / 生成规则和适用范围执行；验证时读回公式文本确认必要的空值保护；除法 / 比率类公式还要确认分母空值和 0 值保护
- 条件格式：完成率 < 80% 标红；状态 = "已完成" 标绿
- 可用性：冻结表头，开启筛选，备注列宽 240px 并自动换行
```

### 11.3 附件来源

若数据来自 Step 2 抽取的附件：

- Columns 的"来源"列写 `reference/<stem>/content.md` 或段落定位。
- 不确定的数据不要强行造，写入 Notes 让子代理按附件核对。
- 如果附件是图片/扫描内容，说明需要读取 `thumbnails/` 辅助判断。

## 12. 禁止事项

- 不要设计超过用户需求的内容。用户要 3 列模板，不要主动加 10 列和多个 sheet。
- 不要在 schema 中写 MCP 工具名或 API 调用；schema 描述"是什么"，不描述"调用什么工具"。
- 不要逐格施工描述，如大量写"A3 写 xxx、B3 写 xxx"；Columns 表和 Notes 已足够表达结构。
- 不要把可计算结果写死；除非用户只要静态展示。
- 不要滥用合并单元格；普通明细表和需要筛选的数据表尤其禁止。
- 不要使用过多颜色；同一 sheet 中主色、浅底、警示色之外不要继续扩展。
- 不要让单个 sheet 超过 1000 行样例数据；如需大量数据，标注"子代理用脚本批量生成"。
- 不要在空模板中显示公式错误值，如 `#DIV/0!`、`#VALUE!`、`#REF!`。
- 不要用真实敏感信息做样例数据。
